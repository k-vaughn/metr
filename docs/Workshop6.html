Vehicle Information Needs

Questions for METR Workshop 6: Vehicle Information Needs

The sixth METR workshop will include focus on the following questions:

  • What data does a vehicle need from METR to properly handle a static operational design domain (ODD) boundary (e.g., an exit from the motorway)?
  • What is the operational scenario associated with a vehicle approaching a static ODD boundary that limits the vehicle’s operation?
  • What additional information needs to be conveyed by METR to enable a vehicle to properly handle a dynamic ODD boundary (e.g., an area experiencing snow)?
  • What is the operational scenario associated with a vehicle encountering a dynamic ODD boundary that limits the vehicle’s operation?
  • How should a METR (esp. an automated) vehicle behave in a region with only partial METR support, should this be a part of the ODD definition?
  • To what extent does METR need to provide warnings, especially to automated vehicles, that you are approaching an area without METR coverage or with a lower level of conformance? Does METR need to identify adjacent jurisdictional areas/METR download zones?
  • While the goal is clearly to avoid conflicts within METR, what should a user (or user system) do when encountering conflicting METR regulations (e.g., from different levels of jurisdictions)?
  • What, if any, attributes (e.g., priority information) need to be conveyed by METR so that users and/or user systems can appropriately process potentially conflicting rules from different layers of jurisdictional entities?
